# 测试点检查项（UI 渲染 + 交互）

> Step 3 拆 UI 渲染组与交互组时加载。挑相关的列成用例，不照单全收。

## UI 渲染点检查项（→ `TC-UI-`）

- **默认态**：进页面第一眼看到什么（扫码面板 vs 账号登录面板）
- **切换态**：点切换后多了 / 少了哪些元素
- **元素可见性**：某元素在该步骤是否存在（如未切账号登录前没有手机号框）
- **文案**：标题、placeholder、提示语、按钮文案是否准确
- **弹窗**：打开 / 关闭 / 多弹窗叠加的作用域
- **新标签页**：弹出后 URL 是否跨域、主标签页是否不受影响
- **列表**：空列表 / 有数据列表是否正常渲染
- **按钮态**：禁用 / 可用 / loading

## 交互点检查项（→ `TC-IX-`）

- 点击 → 跳转（URL 变化；断言 **hostname / pathname**，勿只匹配整串 URL——query 里的 `redirect_uri` 易假通过）
- 点击 → 弹窗（`waitForSelector` 命中）
- 表单填写 → 必填校验、格式校验
- 提交 → 成功反馈（dialog 关闭、列表刷新、成功提示）
- 提交 → 失败反馈（错误提示文案、dialog 不关）
- 跨标签页 / 跨域 session（cookie 是否带过去）
- 去重 / 跳过逻辑（同名是否被跳过）
- 身份 / 角色切换（URL 是否离开切换页）

## 注释里常见该写的「踩坑」

注释讲"意图 + 踩坑"，不是复述步骤。真实例子：

- 文本是「**版块**管理员」不是「板块」
- 「确 **定**」按钮中间有空格
- 「适用范围」input 是 readonly，要 `click({ force: true })`
- 「前往后台」是 SPAN 不是 button
- 组织项是 `<dd>` 不是 button
- 跨 dialog 操作必须 `.el-dialog.first() / .last()` 限定作用域
- `waitForURL(/work-betacloud/)` 会命中 `redirect_uri` query → 假通过；要判 hostname + 排除 `/passport/`
- 完整 skill 脚本勿 `cp` 到 `/tmp` 再 `run.js`（旧 wrap 破坏 `__dirname`）

写注释时问自己：**这条用例如果失败，最可能是踩了哪个坑？** 把那个坑写出来。
