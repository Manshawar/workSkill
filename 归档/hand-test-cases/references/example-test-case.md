# 用例示范（带注释）

> 不确定「注释」字段写多深、预期怎么写才算可判定时加载。

## 示范：以「主题名称」输入框套边界值清单

> 输入框：`input[placeholder="请输入主题名称"]`，`maxlength=20`。

```markdown
## TC-BV-03　主题名称 = 空

**测试点**：不填主题名称直接点「确定」，是否被必填校验拦截。
**类型**：边界值。
**前提**：已打开「新增主题」dialog。
**步骤**：
1. 不填 `input[placeholder="请输入主题名称"]`
2. 勾选适用范围（测试 / 研发）+ 填描述
3. 点 `.el-dialog` first() 的 `button:has-text("确定")`
**预期**：dialog 不关闭，主题名称框出现红色必填提示，列表不新增。
**注释**：必填校验是业务侧硬约束；脚本里默认每次都填了名字，这条专测"漏填"分支，确保前端拦得住、不会建出空名主题。

## TC-BV-04　主题名称 = 21 字（超长）

**测试点**：名称超 maxlength（20）时的表现。
**类型**：边界值。
**前提**：已打开「新增主题」dialog。
**步骤**：
1. 向 `input[placeholder="请输入主题名称"]` 粘贴 21 个字符
2. 观察框内实际内容长度
**预期**：框内只保留前 20 字（前端 maxlength 截断），不报错。
**注释**：maxlength 是前端约束，粘贴超长文本会被静默截断——测的是"用户以为输了 21 字其实只存了 20 字"，避免后续按 21 字去查数据对不上。
```

## 注释写得好 vs 写得差

```markdown
# 差——复述步骤，等于没注释
**注释**：测试主题名称为空时是否报错。

# 好——讲意图 + 踩坑
**注释**：脚本默认每次都填名字，这条专测"漏填"分支；踩坑点：必填提示
是红色文字不是弹窗，别误判成没拦住。
```

## 预期可判定 vs 空话

```markdown
# 差——空话，无法判定
**预期**：页面正常，功能正常。

# 好——可肉眼 / 可截图判定的事实
**预期**：dialog 不关闭，主题名称框下方出现红色"请输入主题名称"，列表行数不变。
```
